# 実験分析ガイド

このドキュメントは、Claude Code実験の分析を統一的な形式で記述するためのガイドラインです。

## 分析ドキュメントの構造

実験分析は以下の構造に従って記述してください。

### 1. 実験日

```markdown
## 実験日

YYYY-MM-DD
```

- 実験を実施した日付を記録
- YYYY-MM-DD形式で記載

### 2. 実験概要

```markdown
## 実験概要

[実験の目的と内容を簡潔に説明]

**タスク内容:**
- [実施したタスク1]
- [実施したタスク2]
- ...

**実験環境:**
- Claude Code バージョン: [バージョン]
- Claude モデル: [モデル名]
```

**記述のポイント:**
- 実験の目的を明確に
- 比較対象がある場合は明示
- 環境情報は再現性のため必須

### 3. 比較結果

実験で比較した項目ごとに、表形式で整理します。

```markdown
### [比較軸1: 例: 対話フローの効率性]

| 項目 | パターンA | パターンB |
|------|-----------|-----------|
| [評価項目1] | [結果A] | [結果B] |
| [評価項目2] | [結果A] | [結果B] |
```

**推奨される比較軸:**

1. **対話フローの効率性**
   - 対話の自然さ
   - やり取り回数
   - セッション継続性

2. **コンテキスト管理**
   - コンテキスト分断の有無
   - 状態保持の方法
   - 暗黙知の保持

3. **成果物の品質**
   - 文書量
   - 詳細度
   - 具体性

4. **コンテキスト消費**
   - トークン使用量
   - Free space
   - 差分と割合

5. **柔軟性**
   - 修正の容易さ
   - 追加確認の可否

**記述のポイント:**
- 記号を活用（◎/○/△/×）
- 数値データは具体的に
- 違いが明確になるように対比

### 4. 設計思想の違い

```markdown
## 設計思想の違い

### [アプローチA]の設計思想
**「[キーフレーズ]」**

メリット:
- [メリット1]
- [メリット2]

デメリット:
- [デメリット1]
- [デメリット2]

**なぜ[特徴]になるのか:**
- [理由1]
- [理由2]
```

**記述のポイント:**
- それぞれの本質的な違いを説明
- 「なぜそうなるのか」という理由を明確に
- メリット・デメリットの両面を記載

### 5. 結論

```markdown
## 結論

### [タスクタイプ]における[アプローチ]の適性

**[適性評価: ◎/○/△/×] [短い要約]**

理由：
1. **[理由1のタイトル]**
   - [詳細1]
   - [詳細2]

2. **[理由2のタイトル]**
   - [詳細1]
   - [詳細2]
```

**記述のポイント:**
- 明確な結論を先に提示
- 理由は番号付きリストで構造化
- 各理由に具体的な説明を追加

### 6. 適用ガイドライン

```markdown
## [アプローチ名]が有効なケース

### ✅ 適しているタスク

**[タスクタイプ1]:**
```
[具体例やフロー図]
```

**特徴:**
- [特徴1]
- [特徴2]

### ❌ 適していないタスク

**[タスクタイプ2]:**
```
[具体例やフロー図]
```

**特徴:**
- [特徴1]
- [特徴2]
```

**記述のポイント:**
- 適用すべき場面を明確に
- 具体例やフロー図で理解しやすく
- 判断基準を明示

### 7. 推奨事項

```markdown
## 推奨事項

### 1. [ユースケース1]

**推奨: [アプローチ名]**

理由:
- [理由1]
- [理由2]

### 2. 一般原則

**[判断基準]:**
```
[判断フロー図]
```
```

**記述のポイント:**
- 実務で使える具体的な推奨
- 判断フローがあると効果的
- 複数のユースケースをカバー

### 8. 今後の検討事項

```markdown
## 今後の検討事項

1. **[検討項目1]**
   - [具体的な内容1]
   - [具体的な内容2]

2. **[検討項目2]**
   - [具体的な内容1]
```

**記述のポイント:**
- 未解決の課題
- 追加検証が必要な仮説
- 改善の可能性

### 9. 参考資料

```markdown
## 参考資料

- [資料1]: [パス/説明]
- [資料2]: [パス/説明]

---

**最終更新**: [日時]
**実験者**: [名前]
```

**記述のポイント:**
- 関連する会話ログ、成果物へのリンク
- 再現に必要な情報
- メタデータ（更新日時、実験者）

## 重要な原則

### 1. 客観性と再現性

- **事実と解釈を明確に分離**
  - 観測された事実は明確に記載
  - 解釈や推論は「これが意味すること」等で区別

- **具体的な数値データ**
  - 可能な限り定量的に記録
  - トークン使用量、文書量、時間など

- **再現可能性**
  - 環境情報を詳細に記録
  - 実験手順を追跡可能に

### 2. 構造化と読みやすさ

- **階層構造**
  - 見出しレベルを適切に使用
  - 情報の階層を明確に

- **視覚的な整理**
  - 表、コードブロック、引用を活用
  - 記号（◎○△×、✅❌）で視認性向上

- **簡潔な文章**
  - 要点を先に
  - 冗長な説明を避ける

### 3. 実用性

- **実務で使える知見**
  - 「いつ使うべきか」を明確に
  - 判断基準を提示

- **具体例の提供**
  - 抽象的な説明だけでなく具体例も
  - フロー図や疑似コードを活用

- **バランスの取れた評価**
  - メリット・デメリット両面を記載
  - トレードオフを明確に

## 参考: exp02の分析例

[exp02-subagent-dialog/analysis.md](../exp02-subagent-dialog/analysis.md)は、このガイドラインに沿った良い例です。
以下の点が優れています：

1. **明確な構造**: 各セクションが明確に分離され、情報が整理されている
2. **具体的な比較**: 表形式で視覚的に比較し、記号で評価を明示
3. **理由の明確化**: 「なぜそうなるのか」を丁寧に説明
4. **実用的な推奨**: 具体的なユースケースと判断フローを提供
5. **フロー図の活用**: コードブロック内のASCIIアートで処理フローを視覚化
